## はじめに

### DriveRev ラボへようこそ！

**DriveRev** は、レンタカーサービスとカスタマーサポートのフロントエンドで、DevRev のワークフローと自動化の包括的なトレーニング用に設計されています。会話型カスタマーエクスペリエンスとカスタマーサポートのユースケースについて、しっかりとした理解を構築できます。

DriveRev サイトは完全に API バックエンドをベースにしており、DevRev との統合が容易です。また、ウェブサイトを完全に再構成する必要なく、独自の DevRev 組織を接続できます。複数のユーザーが同時に異なる構成でソリューションを使用できます。

### DevRev とは？

**DevRev** は、カスタマーサポート（CX）とプロダクト開発を統合した AI ファーストのプラットフォームです。開発チームとカスタマーサポートチームの間のサイロを解消し、顧客フィードバックを製品改善に直接つなげます。

**主な機能：**

- **PLuG（Platform for User Growth）**：顧客とのエンゲージメントを促進する会話型 AI インターフェース
- **AI エージェント**：複雑なワークフローを自動化し、コンテキストに応じた応答を提供
- **統合ナレッジベース**：AI エージェントが活用できる一元化された情報リポジトリ
- **ワークフロー自動化**：カスタムビジネスロジックと外部 API の統合

### なぜ AI エージェントなのか？

従来のチャットボットは、事前にプログラムされた応答に制限されていました。**AI エージェント**は以下を実現します：

- ✅ **自然言語理解**：顧客の意図を理解し、適切に応答
- ✅ **コンテキスト認識**：過去のやりとりとユーザー情報を記憶
- ✅ **動的アクション実行**：予約作成、情報取得、チケット作成などのタスクを実行
- ✅ **継続的学習**：インタラクションとフィードバックから改善
- ✅ **シームレスなエスカレーション**：必要に応じて人間のエージェントに引き継ぎ

### このラボで構築するもの

このラボでは、**DriveRev** のための完全な AI 駆動型カスタマーサポートシステムを構築します：

🚗 **カスタマーサポートエージェント**

- 一般的な質問に回答
- レンタカーサービス情報を提供
- 複雑な問題をエスカレーション

📅 **AI 予約アシスタント**

- 車両の予約をスケジュール
- 利用可能な車両と日程を確認
- 予約の確認とリマインダーを送信

📚 **ナレッジベース統合**

- サービス情報
- 車両詳細
- よくある質問と回答

🔄 **カスタムワークフロー**

- API キー取得

- エージェントへの会話ルーティング

## DevRev 組織の準備

DevRev 組織の基本設定を行い、DriveRev との統合準備を整えます。

### 1.1 PLuG のセットアップ

#### ステップ 1：DevRev にログイン

➔ DevRev アカウントにログインし、DriveRev と統合したい組織を選択します。

> **注意**：DevRev アカウントまたは組織をまだお持ちでない場合は、https://devrev.ai にアクセスしてアカウントと組織を作成できます。

#### ステップ 2：組織アバターの変更

会話で使用されるデフォルトアバターの外観を変更しましょう。

➔ 画面左上のイニシャルの横にある歯車アイコンをクリックして、**Settings** メニューに移動します。

➔ メニューオプションで、**General** オプションをクリックします。

➔ **Edit details** をクリックし、組織名を DriveRev に変更します。

➔ 最後に、**Update** をクリックします。

#### ステップ 3：PLuG Chat App ID を取得

➔ **PLuG & Portal** セクションの下にある **PLuG Chat** メニューオプションを選択します。

➔ Configuration タブで **Your unique app ID** を見つけます。これを後で使用するためにメモにコピーします。

> 💡 **重要**：この App ID は DriveRev との統合に必要です。

#### ステップ 4：ブランドロゴとスタイリング

➔ **Styling タブ**をクリックし、新しい Brand Logo をアップロードします。

➔ 好みに応じて、Appearance と Accent Color も変更してください。これにより、後で DriveRev にリンクされたときに「あなたの」PLuG であることがわかります。

➔ これを保存するには、画面右上の **Save and Publish** ボタンをクリックします。

#### ステップ 5：Help タブを有効化

➔ **Tabs タブ**で、**Help** を有効にします。再度 **Save and Publish** をクリックします。

#### ステップ 6：検索機能の設定

➔ **Layout タブ**で、**Search** ボックスの下向き矢印をクリックします。これによりいくつかの追加オプションが開きます。

➔ **Turing Search** と **Open Articles within PLuG** の両方をオンにします。

➔ **Save and Publish** をクリックします。

#### ステップ 7：PLuG Nudges の設定

PLuG Nudges を追加します。これらは、画面上部に表示されるテキストバナー、または PLuG ランチャーアイコンからアンカーされた画像とテキストを含む完全なカードのいずれかです。これらを使用して、ユーザーアクションを呼び出したり、スケジュールされた間隔で情報インプレッションをスケジュールしたりできます。

➔ **PLuG Nudges** オプションを選択します。

➔ **+Nudge** をクリックします。

**Nudge 1：レンタカーサポート**

➔ タイプ **SpotLight** を選択し、**Create** をクリックします。

➔ Content タブで、上部にタイトルを付けます（鉛筆をクリックして編集）：「_レンタカーサポート_」

➔ タイトル「_レンタカーサポート_」と説明「_車両やサービスについてお困りですか？お問い合わせください！_」を入力します。

➔ Action として **Open widget** を選択します。

➔ Rule タブで、**Page rule** を追加し、URL を「**is**」に設定してから、DriveRev の車両ページの URL を入力します。

➔ 画面右上の **Publish** ボタンをクリックしてこの Nudge を保存します。

**Nudge 2：AI カスタマーサポート**

➔ 2 番目の Spotlight Nudge には、名前「_AI カスタマーサポート_」、タイトル「_AI アシスタントがサポートします_」を使用します。

➔ **Action** は「**Open Widget**」にする必要があります。

➔ **Rules** タブで、URL を DriveRev のホームページ URL に一致するように設定し、**When to send** を _2_（2 秒遅延を示します）に設定します。

➔ Nudge を **Publish** します。

**Nudge 3：予約アシスタント**

➔ 3 番目の Spotlight Nudge には、名前「_予約アシスタント_」、タイトル「_簡単予約！_」を使用します。

➔ **Action** は **Open Widget** にする必要があります。

➔ Rules ページで、URL を DriveRev の予約ページ URL に一致させ、_2_ 秒遅延で **Publish** します。

#### ステップ 8：Application Access Token (AAT) の作成

認証された PLuG ユーザーを持つには、このトークンが必要です。

➔ 左側のメニューで **PLuG Tokens** をクリックします。

➔ Application access tokens ボックス内の **New Token** ボタンをクリックします。

➔ 名前と有効期限を設定し、**Generate** をクリックします。

➔ 生成されたら、キーをメモにコピーしてください。後で必要になるため、二度と表示されません。

> ⚠️ **セキュリティ警告**：AAT は API パスワードのようなものです。絶対に公開リポジトリにコミットしたり、他人と共有しないでください。

#### ステップ 9：Session Replays を有効化

これにより、PLuG を通じて記録されるサイトのセッション分析を有効にできます。

➔ **Session Replays** メニューオプションをクリックします。

➔ **Web** タブを選択し、Session Recordings を **on** に設定します。

➔ **Save and Publish** をクリックします。

**おめでとうございます！PLuG が完全にセットアップされました！**

### 1.2 PLuG 用のエージェントを設定

PLuG が応答できるようにするには、まずエージェントを作成し、次に PLuG の会話をそのエージェントにルーティングするワークフローを作成する必要があります。

#### ステップ 1：エージェントビルダーにアクセス

➔ DevRev 組織にログインし、Settings メニューの **Agents** オプションに移動します。

これが初めて組織を使用する場合、Agent Builder にはまだエージェントがありません。

#### ステップ 2：エージェントを作成

➔ 画面右上の **+Agent Actions** ボタンをクリックします。

➔ アバターをアップロードし、エージェントに名前を付けます：「_カスタマーサポートエージェント_」

➔ 次に、目標を設定します。例：「_顧客向けサポートエージェント。顧客の質問に答え、車両、予約、サービスに関する問い合わせを支援します。_」

➔ 完了したら、**Update Agent** をクリックします。

これで、非常に基本的なプロンプトを持つ初期エージェントのセットアップが完了しました。

#### ステップ 3：基本プロンプトのカスタマイズ

今のところ、後でセットアップをテストできるように、デフォルトのプロンプトに小さな変更を加えます。

➔ **Instructions** ペインをクリックします。選択されると、**Edit Instructions** ボタンが利用可能になります。クリックしてください。

➔ 常に特定の挨拶で開始するように指示する追加の行を追加します。例：「_常に『こんにちは！DriveRev へようこそ！』という挨拶でお客様を迎えてください。_」または他にお好きなものを。

➔ 完了したら、**Save changes** をクリックします。

➔ 右側のテストペインに何か入力して、これをテストできます。今すぐ試してください。

完璧です！エージェントが私たちの指示に従っています！

### 1.4 PLuG から エージェントへの会話ルーティング

最後に行うべきことは、PLuG から発信されるすべての会話が正しいエージェントにルーティングされるようにすることです。

これには、トリガーベースのワークフローを作成します。トリガーは、チケットの作成などの DevRev システムのイベントですが、会話の開始もトリガーです。このイベントを使用して、会話の開始をピックアップし、Talk to Agent ノードにルーティングできます。

#### ステップ 1：ワークフローを作成

➔ **Settings** メニューの **Workflows** オプションに移動します。

➔ 画面右上の **+ Workflow** ボタンをクリックします。

➔ **Build from Scratch** を選択します。

#### ステップ 2：ワークフローの設定

➔ 画面右上の鉛筆アイコンをクリックします。

➔ 名前を _RoutePLuGtoAgent_ に変更します。

➔ 簡単な説明を入力します：「_PLuG の会話をカスタマーサポートエージェントにルーティングします。_」

➔ **Save** をクリックします。

#### ステップ 3：トリガーを追加

➔ キャンバスの **Start** ノードをクリックします。

➔ **Add trigger** をクリックし、**Conversation Created** を選択します。

➔ フィルターで、**Source** を **PLuG** に設定します。これにより、PLuG から発信された会話のみがこのワークフローをトリガーします。

➔ **Save** をクリックします。

#### ステップ 4：Talk to Agent ノードを追加

➔ **Conversation Created** ノードから **+** ボタンをクリックします。

➔ **Talk to Agent** ノードを選択します。

➔ **Agent** フィールドで、先ほど作成した「カスタマーサポートエージェント」を選択します。

➔ **Conversation** フィールドは、トリガーから自動的に渡される会話 ID を使用します。

➔ **Save** をクリックします。

#### ステップ 5：ワークフローを公開

➔ 画面右上の **Publish** ボタンをクリックします。

ワークフローがアクティブになりました！

## DriveRev アカウントセットアップと統合

次に、DevRev 設定を DriveRev アプリケーションに接続します。

### 利用モードの切り替え（Living Docs）

- **ゲストモードの確認**  
  サインアップ前に PLuG ランチャーを開くと、運営側が設定した共有 App ID / AAT を使ったチャットが表示されます。トークン値は公開していませんが、ゲストでも基本動作を試せます。

- **ログイン後の個人設定**  
  プロフィールの **DevRev Integration** で App ID / PAT / AAT を入力し、`個人設定を有効化` をオンにすると、自身の DevRev 組織の PLuG が DriveRev 上で動きます。再度オフにすればゲスト設定へ戻せます。

- **切り替えの目安**  
  PLuG ウィジェットのブランドが変わらない場合は個人設定が未入力か無効です。入力値を確認し、保存後にページを再読み込みしてください。

| よくある症状                              | どうすればよいか                                                |
| ----------------------------------------- | --------------------------------------------------------------- |
| PLuG がゲストのまま表示される            | `個人設定を有効化` がオンか確認し、App ID / AAT を再保存する   |
| 「Session Token 生成に失敗」と表示される | AAT の有効期限切れか入力ミスの可能性。DevRev で AAT を再発行   |
| PLuG ランチャーが表示されない            | `frontend` を再読み込みし、ブラウザのコンソールにエラーがないか確認 |
| 個人設定をオフに戻したい                  | プロフィールで `個人設定を有効化` をオフ → 保存 → ページ再読み込み |

### ステップ 1：DriveRev にサインアップ

1. DriveRev サイトにアクセス
2. **Sign Up** をクリック
3. アカウントを作成：
   - **ユーザー名**：任意のユーザー名
   - **メール**：あなたのメールアドレス
   - **パスワード**：安全なパスワード
   - **フルネーム**：あなたの名前
4. **Create Account** をクリック

### ステップ 2：DevRev 統合を設定

1. DriveRev にログイン後、**User Profile**（右上のプロフィールアイコン）に移動
2. **DevRev Integration** セクションまでスクロール

- **URL**: `https://your-driverev-domain/api/v1/reservations/{reservation_id}`
- **Method**: `GET`
- **Headers**: `Authorization` with API key

➔ **Publish** してエージェントに追加します。

**ワークフロー：車両の現在位置を追跡**

➔ 「_TrackVehicleLocation_」ワークフローを作成します（リアルタイム追跡機能がある場合）。

➔ 同様に設定して Publish します。

### 3.5 パスワードリセット機能を追加

**ワークフロー：パスワードリセット**

➔ 「_ResetPassword_」ワークフローを作成します。

➔ **Agent Skill Trigger** で：

- `email` (string)

➔ **HTTP** ノードで DriveRev のパスワードリセット API を呼び出します。

➔ **Publish** してエージェントに追加します。

➔ エージェントのプロンプトに以下を追加：

```
パスワードリセットが必要な場合：
1. 顧客のメールアドレスを確認
2. ResetPassword スキルを使用
3. リセットリンクがメールで送信されることを伝える
```

## トラブルシューティングとベストプラクティス

### 一般的な問題

#### 問題 1：PLuG ウィジェットが表示されない

**原因**：

- App ID が正しくない
- AAT の有効期限切れ
- スクリプトの読み込みエラー

**解決策**：

1. ブラウザのコンソール（F12）を開いてエラーを確認
2. App ID と AAT を再確認
3. DevRev の設定ページで PLuG が公開されているか確認
4. ネットワークタブでスクリプトが正しく読み込まれているか確認

#### 問題 2：エージェントが応答しない

**原因**：

- ルーティングワークフローが無効
- エージェントがデプロイされていない
- トリガー条件が一致しない

**解決策**：

1. **Workflows** でルーティングワークフローがアクティブか確認
2. エージェントがデプロイされているか確認
3. ワークフローのトリガー条件を確認（Source: PLuG）
4. **Workflow Runs** でエラーログを確認

#### 問題 3：ワークフローが失敗する

**原因**：

- API エンドポイントが正しくない
- 認証トークンの問題
- 入力パラメータの形式エラー

**解決策**：

1. **Workflow Runs** で失敗した実行を確認
2. 各ノードの入力と出力を確認
3. API エンドポイントを Postman などで直接テスト
4. 認証トークンが有効か確認
5. 入力パラメータの形式を確認

#### 問題 4：ナレッジベースの記事がエージェントに反映されない

**原因**：

- 記事がインデックス化されていない
- エージェントにナレッジソースが追加されていない
- 記事の可視性設定が間違っている

**解決策**：

1. 記事が **Published** 状態か確認
2. Visible to が **Customers** に設定されているか確認
3. Agent Builder でナレッジソース（Article）が追加されているか確認
4. 数分待ってからテスト（インデックス化には時間がかかる場合があります）

### ベストプラクティス

#### プロンプト設計

1. **明確で具体的に**：エージェントの役割と責任を明確に定義
2. **例を提供**：望ましい会話フローの例を含める
3. **境界を設定**：エージェントが何をすべきでないかを明示
4. **反復的に改善**：実際の会話をもとにプロンプトを継続的に改善

#### ワークフロー設計

1. **単一責任**：各ワークフローは 1 つの明確な目的を持つ
2. **エラーハンドリング**：失敗した場合の処理を含める
3. **ログとデバッグ**：重要なデータポイントをログに記録
4. **テストと検証**：本番環境にデプロイする前に徹底的にテスト

#### ナレッジベース管理

1. **定期的な更新**：情報が常に最新であることを確認
2. **構造化された内容**：見出し、リスト、表を使用して読みやすく
3. **検索最適化**：重要なキーワードを含める
4. **カテゴリー分類**：関連記事をグループ化

#### セキュリティ

1. **トークン管理**：AAT と API キーを安全に保管
2. **最小権限の原則**：必要な権限のみを付与
3. **定期的なレビュー**：アクセス権限を定期的に確認
4. **監査ログ**：重要なアクションをログに記録

### パフォーマンス最適化

1. **レスポンス時間**：ワークフローを最適化して応答時間を短縮
2. **キャッシング**：頻繁にアクセスされるデータをキャッシュ
3. **並列処理**：可能な場合は複数のタスクを並列実行
4. **モニタリング**：パフォーマンスメトリクスを継続的に監視
